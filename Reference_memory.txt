Named targets (defined in workflow/Snakefile — see section 2):
  all          — default target if none given: preprocess + build, every
                 discovered basin (rules 01-16)
  preprocess   — rules 01-12 only (everything up to and including the
                 upstream-boundary check), every discovered basin
  build        — rules 13-16 only (SFINCS build, spin-up, sanity checks,
                 main event run) — assumes preprocess outputs already exist

Key flags:
  --cores N          Required. Max parallel jobs. N=1 is fine for a single
                      basin (the DAG is a long linear chain per basin
                      anyway); use a higher N when running multiple basins
                      so independent basins' jobs can run concurrently.
  -n, --dry-run       Preview the DAG (which jobs, in what order, and WHY
                      each is scheduled) without executing anything. Always
                      dry-run after editing rules/config before a real run.
  --forceall          Force every scheduled job to re-execute even if its
                      outputs already exist and look up to date. Combine
                      with a restricted target (see target_basins below) to
                      force-rerun just one basin without touching others.
  --forcerun RULE     Force only a specific named rule (and everything
                      downstream of it) to rerun, leaving unrelated
                      up-to-date outputs alone — narrower than --forceall.
  --until RULE        Run the DAG only up to and including RULE, then stop
                      (e.g. --until build_sfincs to assemble the SFINCS
                      input files without actually executing the solver).
  --config KEY=VALUE  Override/extend a config.yml value from the CLI for
                      this run only (parsed as YAML, so lists/numbers work:
                      --config target_basins="[2433835]"). Does not modify
                      config.yml on disk. See "RUNNING A SINGLE BASIN" in
                      section 14 for the basin-restriction use of this.
  --rerun-triggers mtime
                      By default Snakemake also reruns a job when its CODE
                      or PARAMS changed since the last run (provenance
                      tracking), even if the output file's mtime looks
                      current — this is usually what you want, but if a
                      formatting-only script edit is triggering unwanted
                      reruns, this flag switches to mtime-only triggering.
                      (Alternative for one specific output file: snakemake
                      --cleanup-metadata <outfile> wipes its provenance
                      record without touching the file itself.)
  --keep-going        Continue running other, independent jobs if one job
                      fails, instead of stopping the whole run immediately —
                      useful when running several basins at once and one
                      basin's data has a problem.
  --unlock            Release a stale ".snakemake/locks" lock left behind
                      by a crashed/killed run before you can run again.
  -p, --printshellcmds
                      Print the actual shell command each job runs (mainly
                      useful for the SFINCS solver invocations in rules 14/16).
  --rulegraph / --dag
                      Print the DAG in Graphviz dot format (pipe to `dot -Tpng`
                      to render) — needs graphviz installed.

Everyday examples:
  Dry-run everything:
    snakemake -n --cores 1
  Dry-run just the build stage:
    snakemake -n build --cores 1
  Force a full real rerun of ONE basin only (see section 14 for how
  target_basins works and how BASINS is normally discovered):
    snakemake build --cores 1 --config target_basins="[2433835]" --forceall
  Rerun only from build_sfincs onward (e.g. after a config.yml change to
  `sfincs`) without redoing preprocessing:
    snakemake build --cores 1 --forcerun build_sfincs

  03_land_polygons.smk → 04_protection_levels.smk →
  05_static_terrain_data.smk → 06_river_network.smk →
  07_boundary_forcings.smk → 08_clean_river_network.smk →
  09a_river_depth.smk → 09b_estuarine_depth.smk → 10_condition_elevation.smk →
  11_river_preburn.smk → 11b_burn_river_dem.smk → 12_testing.smk →
  13_build_sfincs.smk → 14_run_spinup.smk → 15_sanity_checks.smk →
  16_run_event.smk

Scripts:      workflow/scripts/01_*.py … 16_*.py (mirrors rule-file numbering,
              including the 05a/05b/05c and 09a/09b sub-letters below)
Shared src:   workflow/src/  — see section 9 for the full module list

DAG shape, not a strictly linear chain:
  - Rules 05a/05b/05c (inside 05_static_terrain_data.smk) are a genuinely
    linear sub-chain — elevation → landuse/zsini → roughness, each
    reprojecting onto the previous step's exact grid. This is the one place
    letter suffixes represent a strict sequence rather than a fork.
  - Rules 09a and 09b form an UNCONDITIONAL SEQUENTIAL CHAIN, not a fork: 09b
    (add_estuarine_depth) always runs immediately after 09a (add_river_depth)
    regardless of river_processing.estuarine_depth.enabled, and together they
    produce river_network_estuarine.gpkg — the one "final" network every
    downstream rule uses.
  - Rule 10 (condition_elevation, rule name enforce_river_monotonicity) is
    the ACTUAL independent branch, parallel to the 09a→09b chain: it depends
    on 09a's output too, but not on 09b's. Rule 11 (river_preburn, rule name
    burn_river_bed) is the merge point needing both 09b's estuarine-adjusted
    network AND 10's conditioned elevation.
  - Rule 11b (burn_river_dem, opt-in — river_processing.burn_rivers.enabled,
    requires conditioning.enabled) runs right after rule 11: burns
    zbed_anchors.gpkg into a channel-only, native-resolution DEM
    (river_burned_dem.tif), fed into build_sfincs (13) as a higher-priority
    elevation source. See section 5.
  - Rule 12 (testing) is a diagnostic-only side branch containing TWO rules:
    test_upstream_boundary (needs rules 07/09b's outputs) and
    test_bifurcation_calibration_options (needs rule 07's river_forcing.nc +
    the raw SWORD catalogue sources directly). Neither gates build_sfincs
    (13), and nothing downstream depends on either's output.
  - Rule 07 (get_boundary_forcings) sits between the raw network (06) and the
    cleaned network (08) because rule 08 genuinely needs rule 07's output
    (the boundary crossing points) to seed its BFS cleaning — rule 07 also
    builds surge_forcing.nc, unrelated to the river network, so it cannot be
    collapsed into a 06-family sub-letter.

Named aggregate targets in Snakefile:
  - `preprocess`  → _PREPROCESS_OUTPUTS (rules 01–12)
  - `build`       → _BUILD_OUTPUTS (rules 13–16)
  - `all`         → both (default target)

_PREPROCESS_OUTPUTS / _BUILD_OUTPUTS are the canonical output lists; any new
rule output that should be part of the pipeline MUST be added there.
Config-conditional blocks:
  - river-conditioning outputs (elevation_conditioned.tif, zbed_anchors.gpkg,
    rules 10/11 plots) only if river_processing.conditioning.enabled
  - river_burned_dem.tif + its plot (rule 11b) only if
    river_processing.burn_rivers.enabled (requires conditioning.enabled too,
    validated in 00_common.smk)
  - sfincs_subgrid.nc only if sfincs.subgrid.enabled
  - refinement_polygons.gpkg + plot only if sfincs.grid.quadtree.enabled
  - spinup + sanity-check outputs (incl. velocity animation), AND rule 16's
    (run_event) outputs, only if sfincs.spinup.enabled — rule 16 needs rule
    14's restart file, so it shares that gate.
  river_network_estuarine.gpkg and its 09b hydraulics plot are ALWAYS
  produced (add_estuarine_depth always runs after add_river_depth — a
  disabled estuarine model just means a passthrough copy with power-law
  depths only).

RULE SUMMARY (in DAG order):
  00  common.smk     — catalogue helper (CATALOGUE / catalogue_path), wildcard
                        constraints, BASINS list, config validation (quadtree
                        requires subgrid; boundary_setup.mode enum check).
                        BASINS is auto-discovered from delta_polygons but can
                        be restricted via an optional CLI-only config
                        override, target_basins (not a config.yml key) — see
                        "RUNNING A SINGLE BASIN" below.
  01  separate_deltas — clip delta polygon per basin_id (BasinID2)
  02  modeldomain    — domain bbox, UTM CRS, domain.gpkg. Domain is ALWAYS
                        the delta polygon itself (reprojected to
                        auto-detected UTM) — the old "basins" mode (bbox
                        around intersecting HydroBASINS) was removed
                        entirely. {basin_id}_intersecting_basins.gpkg is
                        still written (schema compatibility) but is now just
                        a copy of the domain polygon, n_intersecting_basins
                        hardcoded to 0.
  03  get_land_polygons — clip OSM land polygons to domain. Runs right after
                        domain (02) because it's a PREREQUISITE of elevation
                        (05a), landuse (05b), and river network (06). Also
                        used by build_sfincs (13) to define waterlevel
                        boundary cells: active-domain edge cells NOT covered
                        by land polygons become mask=2 (waterlevel boundary).
  04  get_protection_levels — FLOPROS × WRI geounit dominant protection
                        standard per delta polygon, split by riverine/coastal
                        hazard. Depends ONLY on the delta polygon (01) —
                        unrelated to elevation/landuse/terrain. Always runs;
                        protection_levels.enabled only gates whether rule 07
                        subtracts the identified level from the forcing.
  05a get_elevation    — elevation_merged.tif (FathomDEM + GEBCO merge, see
                        section 4). Genuinely sequential with 05b/05c.
  05b get_landuse      — landuse.tif; also builds the full zsini.tif (needs
                        05a's elevation_merged.tif + land_polygons).
  05c get_roughness    — roughness.tif (needs 05b's landuse.tif).
  06  get_river_network — river_network.gpkg; clips SWORD to the domain and
                        writes it straight out. Needs land polygons (03) +
                        landuse (05b, for the plot's water-body overlay only)
                        — does NOT read elevation (05a) or perform any
                        elevation sampling/monotonicity enforcement anymore
                        (that logic, and the max_elevation attribute it used
                        to produce, were removed — see section 6).
  07  get_boundary_forcings — surge (CoastRP EVA) + river (GloFAS EVA)
                        forcing NetCDFs, GRDC bias correction, protection-
                        level netting (uses 04's output). Needs 06's raw
                        (uncleaned) river network — this rule's output
                        (crossing points) is what rule 08 cleans FROM.
  08  clean_river_network — BFS from boundary crossings (07's output) →
                        river_network_clean.gpkg; width normalization.
  09a add_river_depth  — Leopold-Maddock hydraulic geometry →
                        river_network_processed.gpkg.
  09b add_estuarine_depth — ALWAYS runs immediately after 09a (unconditional
                        sequence, not a fork). Leuven et al. (2018) estuarine
                        depth blended with power-law depth near the coast
                        (gated internally by river_processing.
                        estuarine_depth.enabled) → river_network_estuarine.gpkg
                        (used by ALL downstream rules); also owns the
                        hydraulics-summary plot.
  10  condition_elevation (opt-in, gate: river_processing.conditioning.
                        enabled; rule name enforce_river_monotonicity) —
                        the independent branch, parallel to the 09a→09b
                        chain (depends on 09a's output, not 09b's). Enforces
                        monotonic non-increasing DEM elevation along every
                        river centerline (downstream direction) →
                        elevation_conditioned.tif. This rule modifies the
                        ELEVATION RASTER, not the river network — the
                        network is only used as a walk guide. It cannot run
                        earlier than it does despite operating on elevation,
                        because it needs 09a's discharge-filtered FINAL
                        reach set, not just the raw network.
  11  river_preburn (opt-in, same gate as 10; rule name burn_river_bed) —
                        MERGE POINT of 09b + 10: samples DEM-pixel-spaced
                        river-bed elevation points along centerlines (needs
                        09b's estuarine-adjusted network AND 10's
                        conditioned elevation) → zbed_anchors.gpkg, feeds
                        SFINCS subgrid burning (gdf_zb) in rule 13.
  11b burn_river_dem (opt-in — river_processing.burn_rivers.enabled, needs
                        conditioning.enabled) — burns zbed_anchors.gpkg
                        directly into a channel-only DEM at native
                        resolution, per-reach (own points + own centerline
                        only, see src/river_burn.py), avoiding a confirmed
                        hydromt_sfincs cross-tile-contamination bug in its
                        own burn_river_rect. Output river_burned_dem.tif is
                        registered in rule 13 as a HIGHER-PRIORITY elevation
                        source ahead of elevation_merged/
                        elevation_conditioned (first-source-wins merge;
                        gaps still fall back to the existing DEM), and
                        burn_river_rect is skipped entirely when this is
                        enabled (no double-burning). See section 5.
  12  testing         — TWO diagnostic-only side-branch rules, neither gates
                        build_sfincs (13):
                          test_upstream_boundary — wave-propagation check
                            (needs 07 + 09b's outputs — see section 8).
                          test_bifurcation_calibration_options — compares
                            discharge partitioning at every bifurcation
                            between the original and manually-corrected
                            SWORD sources (needs 07's river_forcing.nc +
                            the raw SWORD catalogue entries directly, not
                            rule 06's output — see section 8).
  13  build_sfincs    — HydroMT-SFINCS model build (regular or quadtree grid,
                        see section 10); elevation source priority:
                        elevation_conditioned.tif (10) > elevation_merged.tif
                        (05a). Only ASSEMBLES config/input files — does NOT
                        execute the SFINCS solver (that happens in rule 14).
  14  run_spinup      — runs a short SFINCS spin-up simulation (the solver's
                        FIRST execution in the pipeline) to produce a
                        restart file (grid-type agnostic).
  15  sanity_checks   — post-run inundation ratio checks + flood/velocity
                        animations (grid-type agnostic; SPIN-UP output only).
  16  run_event       — runs the MAIN flood-event simulation: the sfincs.inp
                        rule 13 already wrote (tstart = spin-up end,
                        tstop = end of the full forcing timeseries,
                        rstfile = rule 14's restart file) executed directly
                        in sfincs_root (not a subdirectory). Produces the
                        event's own sanity-check-style diagnostics
                        (inundation-ratio map, flood/velocity animations)
                        plus a per-timestep flooded-area/flood-volume CSV.

5. TERRAIN & DATUM CORRECTION (rule 05a) + ELEVATION CONDITIONING (rule 10)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

FathomDEM is the only coastal DEM source, with no elevation clipping (it is
not restricted to low-lying coastal areas). Vertical datum correction is
mandatory, not optional.

Pipeline in 05a_get_elevation.py:
  1.  Merge FathomDEM tiles for the domain → topo_utm (UTM, coverage-weighted
      "nan-aware" resampling; /100 for the raw centimeter storage).
  2.  Vertical datum correction, EGM2008 → GOCO06s (MANDATORY): add the geoid
      offset N_EGM2008 − N_GOCO06s (pyshtools synthesis) to every valid
      FathomDEM pixel. No MDT subtraction is applied to the DEM itself — only
      the geoid offset.
  3.  Clip GEBCO to the domain, resample to the working UTM grid, and
      re-reference it to GOCO06s by subtracting the MDT (HYBRID-CNES-CLS2022)
      — extrapolated over land via inverse-distance weighting before
      resampling (needed so the subtraction doesn't NaN-poison nearshore
      GEBCO pixels whose receptive field straddles a land-side nodata cell).
      Applied to EVERY valid GEBCO pixel (not just at sea), since GEBCO may
      also fill land-side gaps in step 4.
  4.  HARD MERGE (no land-polygon mask, no gradient blend): merged =
      FathomDEM wherever it has valid data, GEBCO everywhere else. Purely
      driven by FathomDEM's own nodata pattern, not an independent land/sea
      classification. A defensive fillnodata covers the rare case where
      neither source has data for a domain pixel.
  5.  Write elevation_merged.tif (the only raster 05a writes).
  6.  Diagnostic plots (all always produced — datum correction is
      mandatory): 05a_elevation.png (main elevation map),
      05a_elevation_geoid_offset.png, 05a_elevation_mdt_ocean.png,
      05a_elevation_datum_correction.png (DEM correction delta),
      05a_elevation_gebco_correction.png (GEBCO correction delta).

zsini.tif is built in rule get_landuse (05b), not 05a — 05b already needs
land_polygons + elevation_merged for its own reprojection step, and zsini
genuinely needs the land/water-body split. 05b_get_landuse.py: reprojects
landuse onto elevation_merged.tif's grid, then rasterizes land_polygons onto
that same grid to get the initial land/sea split (0.0 m at sea, nodata on
land), overrides landuse==200 (water body) cells to 0.0, and excludes any
cell with no elevation data. Plots: 05b_landuse.png, 05b_zsini.png.

ELEVATION CONDITIONING (rule 10, opt-in — river_processing.conditioning.
enabled; rule name enforce_river_monotonicity):
  This rule modifies the DEM (not the river network) — it conditions the
  ELEVATION RASTER, using the river network purely as a guide for where and
  in what order to walk. enforce_river_monotonicity() in
  src/river_conditioning.py: topological (Kahn's algorithm) walk over the
  FULLY-PROCESSED river network (river_network_processed.gpkg, rule 09a's
  output — deliberately NOT the raw or cleaned network, so conditioning only
  touches reaches that survive 09a's discharge filter and will actually be
  part of the final model), enforcing a per-reach running-minimum along
  centerline pixels so DEM elevation never increases downstream.
  Bifurcations inherit the min of all upstream parents; headwaters seed from
  their own first valid pixel. This is why the rule can't run any earlier
  than it does (a natural question to ask, since it operates on elevation):
  it needs the FINAL reach set, which isn't known until rule 09a completes.
  Output: elevation_conditioned.tif, which feeds directly into build_sfincs
  (13) — elevation source priority is elevation_conditioned.tif (10) >
  elevation_merged.tif (05a). INDEPENDENT of rule 09b (add_estuarine_depth)
  — both only depend on 09a's output, neither depends on the other.
  Uses the shared _sample_line_cells() helper, defined in
  src/river_network.py.
  Diagnostic plot: 10_condition_elevation.png (n_seeds × 2 grid of
  before/after downstream elevation profiles per seed reach).

RIVER-BED PREBURN (rule 11, opt-in — same gate as 10, MERGE POINT of 09b+10):
  compute_river_bed_points() in src/river_preburn.py samples a point at every
  DEM-pixel step along each centerline (via the same _sample_line_cells());
  rivbed = DEM − rivdph (rivdph constant per reach; no re-monotonicity
  enforcement needed since DEM is already conditioned).
  Output zbed_anchors.gpkg feeds sf.subgrid.create(gdf_zb=...) in rule 13 —
  the DEM raster itself is never modified; only the subgrid table is burned
  along the river. Diagnostic plot: 11_river_preburn.png.

RIVER DEM BURNING (rule 11b burn_river_dem, opt-in — river_processing.
burn_rivers.enabled, requires conditioning.enabled — NEW):
  Unlike rule 11 (which only feeds hydromt_sfincs's OWN subgrid burning via
  gdf_zb, not touching the DEM raster), this rule actually WRITES a new
  channel-only DEM raster (river_burned_dem.tif) at native FathomDEM
  resolution. Exists because hydromt_sfincs's own burn_river_rect (invoked
  via subgrid_component.create(river_list=[...]) in rule 13) processes the
  subgrid domain tile-by-tile, matching each tile's local river line against
  the GLOBAL, un-clipped zbed_anchors points with NO distance cutoff —
  confirmed (basin 4267691, Mississippi headwater reach) to turn a near-flat
  ~11.68 m zbed_anchors profile into a wavy, non-monotonic ~12.6-14.8 m
  burned bed level that doesn't track the source data at all.
  burn_river_channel() (src/river_burn.py) burns each reach independently,
  using ONLY that reach's own zbed_anchors points and its own full
  centerline (no cross-reach/cross-tile contamination possible by
  construction). At each reach-to-reach junction, a synthetic boundary
  anchor point (_junction_value()) is inserted at along=0/along=length,
  valued as the AVERAGE of the immediate upstream/downstream neighbour(s)'
  own nearest anchor (averaged across all neighbours at a confluence or
  bifurcation) — this removes what would otherwise be a flat-clamp
  discontinuity where two reaches meet (verified via a synthetic 2-reach
  test: both reaches converge to the identical shared value at their
  junction, not two different values on either side — an earlier, wrong
  first attempt had each side adopt the OTHER side's raw value, which just
  swaps which side shows the mismatch rather than removing it).
  Extrapolation beyond a reach's own anchor range is CLAMPED (not linear),
  to avoid overshoot (observed: basin 2433835 reach 21604100033, a linear
  extrapolation gave -3.9 m vs. -2.4 m real).
  Output river_burned_dem.tif is registered in rule 13 as a HIGHER-PRIORITY
  elevation_list entry ahead of elevation_merged/elevation_conditioned
  (first-source-wins merge — gaps in the channel-only raster still fall
  back to the existing DEM for everywhere else: ocean, floodplain), and
  burn_river_rect is skipped entirely (river_list=[]) when this is enabled,
  to avoid double-burning. Diagnostic plot: 11b_river_burn.png.

Rule 09a (add_river_depth):

  - Bifurcation split: PURE WIDTH-PROPORTIONAL (weight = width). The old

    straight → 1.0, 90° perpendicular → 0.1, linear between) was REMOVED

  - Hydraulic depth from Leopold-Maddock: depth = alpha × Q^beta / width

Rule 09b (add_estuarine_depth) — ALWAYS scheduled immediately after 09a,
regardless of river_processing.estuarine_depth.enabled; an UNCONDITIONAL
SEQUENCE with 09a, not a fork (rule 10 is the actual independent branch —
see section 2):
  - If enabled: loads Nienhuis 72-delta table (load_nienhuis), matches
    basin→delta by spatial proximity (match_basin_to_delta, bbox + buffer),
    then compute_estuarine_depths() — O'Brien (1969) mouth cross-sectional
    area A0 = C·P^alpha, convergence length L_A = L_e / convergence_ratio_k,
    depth d(x) = A0·exp(-x/L_A) / W_total_mouth (normalised by summed
    mouth-reach width, end_reach==2), linear blend zone (±blend_fraction of
    L_e) around the estuarine/fluvial boundary with the power-law rivdph,
    floored at min_depth_m. New columns: rivdph_estuarine, rivdph_powerlaw,
    rivdph_blend_alpha.
  - If disabled or no Nienhuis delta matched within max_match_dist_km:
    passthrough copy (power-law rivdph unchanged for all/some reaches).
  - NEW, applied UNCONDITIONALLY regardless of estuarine_depth.enabled:
    enforce_mouth_depth_monotonic() (src/estuarine_depth.py) floors a
    mouth's final depth at max(own power-law estimate, upstream neighbour's
    depth) — mirrors enforce_mouth_width_monotonic (rule 08) for depth.

RIVER FORCING steps:
  Step 1:  Identify reaches that cross the domain grid boundary (rule 06's
           output)
  Step 2:  resolve_inside_domain_reaches() — clip the crossing reach's own
           geometry against the domain polygon and use the point where it
           first enters the domain (almost always the same point Step 1
           already found). Only walks to the next downstream reach via
           rch_id_dn when the crossing reach merely touches the domain
           boundary with none of its length actually inside it.
  Step 3:  Filter — enters_domain (at least one rch_id_dn listed reach is in
           the domain network)
  Step 4:  visible_on_grid (inside-domain reach width ≥ SFINCS main-grid
           resolution) is now INFORMATIONAL ONLY — it no longer FILTERS out
           a crossing from Step 5 onward. Previously a narrow crossing at
           fine resolution could strand and silently drop a whole
           downstream branch; has_glofas (which crossings actually get
           forcing) now depends only on enters_domain AND EVA convergence.
           Also: sfincs_res (used for this diagnostic) now uses the MAIN
           grid resolution only — the old subgrid-pixel-count adjustment
           was removed, since subgrid doesn't change what resolution SFINCS
           actually solves hydrodynamics on.
  Step 5:  Snap to GloFAS cell (glofas_search_radius_km: 5.0), filtered by
           glofas_min_mean_discharge. glofas_variable is hardcoded "dis24"
           (the sole variable in this catalogue source) — the old
           glofas_variable config key was removed.
  Step 5b: GRDC correlation diagnostics + optional bias correction (per
           matched GloFAS cell, cached in bias_cache like eva_cache):
           - find_nearest_grdc_station() within
             bias_correction.grdc_search_radius_km (15 km — MOVED here from
             a `river`-top-level key; also used by the always-on correlation
             diagnostic below, not just the opt-in bias correction it now
             lives under) of the crossing's inside-domain point.
           - If found: load_grdc_series() + compute_grdc_correlation() —
             always computes GloFAS/GRDC overlap correlation diagnostics,
             INDEPENDENT of bias_correction.enabled. Requires min_overlap_days
             (730 d / ~2 yr); below that, or if no station within radius,
             returns None (logged).
           - TAIL-FOCUSED CORRELATION: correlation_raw (and, from
             bias_correct_discharge(), correlation_corrected) are restricted
             to overlap days where GRDC discharge is at/above
             bias_correction.tail_percentile (default 90.0) of its own
             overlap distribution — NOT the all-days Pearson r — since this
             pipeline only cares about GloFAS/GRDC agreement at
             flood-relevant high flows. The all-days correlation is still
             computed and returned as correlation_full_raw /
             correlation_full_corrected, for reference/plotting context
             (shown alongside the tail r in plot_bias_correction and
             plot_grdc_overview's scatter panels, with tail days
             highlighted). This is the only place tail_percentile is
             configured (boundary_forcings.river.bias_correction).

             if bias_correction.enabled is True (currently FALSE by default)
             AND correlation diagnostics were returned, runs
             empirical_quantile_map() (EQM, n_quantiles=200) with LINEAR TAIL
             EXTRAPOLATION. Otherwise GloFAS series unchanged, diagnostics=None.
           - TODO (in config.yml and the script): the current GRDC extract
             covers only the 8 basins in the 8_polygons.json delta-polygon
             test set (Mississippi, Zambezi, Ebro, Calvert, Mangoky, Vijose,
             Taiwan, Rhine-Meuse) — a global GRDC extract would let this run

             falling back to GloFAS-as-truth.

           discharge (AMAX/GEV fit, unchanged) + a FULL discharge

REDESIGNED EVA OUTPUT — return-period table, build-time discharge (NEW):
  Previously, Step 6 fit POT/GPD and immediately extracted ONE scalar
  (flood_discharge at eva.rp_fl), baked directly into a sinusoidal discharge
  timeseries at THIS rule's own run time (build_river_dataset). Changing the
  design return period meant re-running the whole EVA fit.
  Now: gpd_return_value_table() (src/extreme_values.py, vectorized sibling
  of the existing gpd_return_value() — same fit, no re-fitting) evaluates
  the already-fitted POT/GPD curve at a fixed standard set of return
  periods, STANDARD_RETURN_PERIODS_YR = 1, 1.5, 2, then 5-yr steps to 1000,
  then 1000-yr steps to 10000 (212 values), producing discharge_rp_table
  (dims crossing × return_period) — written to river_forcing.nc alongside
  bankfull_discharge and protection_discharge (scalars, as before) and new
  dataset attrs lead_days/period_hr (so rule 13 can reconstruct the wave
  shape without extra config plumbing).
  build_river_dataset() (river_forcing.py) no longer builds a "discharge"
  timeseries variable at all — sinusoidal_wave() is no longer called here.
  The actual design discharge is looked up AT SFINCS BUILD TIME (rule 13)
  via new build_design_discharge_matrix(river_ds, active, design_rp_yr)
  (river_forcing.py): log-RP-interpolates design_q from the table
  (np.interp on log(RP), standard flood-frequency convention, well-behaved
  across the table's 4-orders-of-magnitude RP span), applies the
  protection-discharge floor piecewise (overtopped → bankfull_q + (design_q
  - protection_q); contained-but-above-bankfull → bankfull_q; else
  unchanged) per crossing, then reconstructs the hydrograph via
  sinusoidal_wave(). boundary_setup.design_rp_river_yr (default 150) drives
  this lookup, replacing the removed boundary_forcings.river.eva.rp_fl.
  CONSEQUENCE: changing design_rp_river_yr now only requires re-running
  rule 13 (build_sfincs), NOT rule 07 (get_boundary_forcings/EVA) — a much
  cheaper iteration loop for calibrating the design RP.
  rp_fl is still read by analyse_cell() internally (unchanged signature),
  but 07_get_boundary_forcings.py now INJECTS design_rp_river_yr into
  eva_cfg["rp_fl"] before calling it — purely so the EVA diagnostic plot's
  vertical line/CI reflect the actual production design RP; it has zero
  effect on the production discharge_rp_table (which is computed at every
  standard RP regardless of rp_fl).
  Three test scripts (test_discharge_sensitivity.py,
  test_discharge_return_period_response.py, test_river_burning_methods.py)
  now call build_design_discharge_matrix directly instead of each
  duplicating GPD-reconstruction logic (deleted their own
  target_discharge()/ad-hoc build_scaled_dis_table() implementations).
  EVA diagnostic-plot crossing selection also changed: now picks the
  crossing with the highest q_rp100 (previously: widest crossing by raw
  SWORD width, which didn't reliably track actual discharge magnitude), and
  reuses the actual (possibly GRDC-bias-corrected) series the production
  numbers were fit on instead of re-fetching raw GloFAS.

REDESIGNED distance formula (previously a single kinematic estimate only):
  final_distance = min(kinematic_distance, attenuation_distance)
    kinematic_distance   = T_eff × (celerity − v_river)
      T_eff    = effective_period_fraction × surge_period_hr × 3600 s
      celerity = sqrt(g × (depth_mouth + surge_amplitude))
      v_river  = Q_bf / (W × D)   [depth_mouth = rivdph]
    attenuation_distance = NEW — marches upstream reach-by-reach (following
      the mainstem, i.e. the largest bankfull_discharge_acc branch at each
      confluence) until friction-damped surge amplitude decays below
      amplitude_threshold_fraction of its own starting value:
        A(x) = A0 · exp(-μx),  μ = r / (2c),
        r = (8/3π) · g · n² · U0 / h^(4/3)
      channel Manning's n = testing.upstream_boundary_check.channel_manning_n.
      U0 (reference velocity) is computed ONCE at the mouth and held FIXED
      for the whole march — re-deriving U0 from the locally-decaying
      amplitude at each step would make the decay non-convergent
      (hyperbolic, not exponential).
  depth_mouth and v_river are used exactly as calculated -- no config floor
  (the old min_depth_at_mouth_m / min_river_velocity_ms were removed
  earlier; amplitude_threshold_fraction is the newest replacement/addition
  for the attenuation branch specifically). A mouth with no usable
  calculated depth/velocity is skipped entirely (logged), not substituted.
  Each mouth now uses its OWN NEAREST CoastRP station's rp_level as surge
  amplitude (not the domain-wide max — stations can vary 5-10x across one
  basin).
  Forcing points must be OUTSIDE the (now-tighter-of-two) distance from
  river mouths. Blocked mouths (v ≥ celerity, kinematic branch
  undetermined): drawn as orange X markers.
  Config params (testing.upstream_boundary_check): effective_period_fraction
  (0.5), channel_manning_n (NEW), amplitude_threshold_fraction (NEW).
  Plot title reports the tightest margin (distance-from-mouth minus the
  final min() distance) found across all forcing-point/mouth pairs; subtitle
  "min depth"/"min v" are the smallest CALCULATED values found. Plot gains a
  LineCollection overlay coloring reaches by remaining amplitude along the
  attenuation march (new), on top of the existing geodesic-circle +
  river-network base:
        Green dot = forcing point outside the critical radius (safe)
        Red dot   = forcing point inside the critical radius (too close)

Observation points (section 11): for each of the N_TOP_CROSSINGS
(sfincs.observation_points.n_top_crossings) boundary crossings with the
highest bankfull discharge, place N_OBS_PER_CROSSING
(observation_points.n_per_crossing) evenly-spaced observation points along
the downstream river path, up to max_downstream_hops reaches.

Water-level boundary forcing (section 9): sf.water_level.create() spatially
matches each surge station to the nearest boundary cell (mask=2) within
boundary_setup.waterlevel_buffer_m. In forcing_mode="river_only", a
flat constant boundary at baseline_m is used instead (the same value
zsini's sea cells carry), via sf.water_level.create_boundary_points_from_
mask(bnd_dist=flat_boundary_point_spacing_m). In forcing_mode="compound"
with boundary_setup.compound.lag_hr != 0, the river discharge timeseries
(section 10) is shifted relative to this surge timeseries before either is
handed to SFINCS — see the config-schema entry above for the exact mechanism.

14_run_spinup.py / 15_sanity_checks.py / 16_run_event.py: all three are
grid-type agnostic. Rule 14 executes SFINCS via subprocess with the working
directory set to spinup/ so all relative paths in the spinup .inp resolve
correctly; the restart filename follows SFINCS v2.3 naming
(sfincs.YYYYMMDD.HHMMSS.rst, timestamp = tref + spinup_days), computed in
14_run_spinup.smk as RST_FNAME so Snakemake knows exactly which file to
expect (also referenced directly in 16_run_event.smk's input:, since
16_run_event.smk is included after 14_run_spinup.smk and Snakemake's
include: shares one global namespace).

16_run_event.py executes SFINCS directly in sfincs_root itself (NOT a
subdirectory) — unlike rule 14, it does not write its own sfincs.inp; rule
13's main sfincs.inp is already fully configured for this exact run
(tstart = spin-up end via "Phase 2 config" in 13_build_sfincs.py, tstop =
end of the forcing timeseries, rstfile = the spin-up restart file). This
produces results/{basin_id}/sfincs/sfincs_map.nc (the event's own map
output, alongside — not inside — spinup/'s own sfincs_map.nc). Diagnostics
mirror rule 15's Check 1 (compute_max_inundation + plot_inundation_check,
zsmax envelope) and the flood-progression animation (compute_flood_
progression + animate_flood_progression), sourced from run_dir=sfincs_root
instead of spinup_dir, written to visuals/model_runs/main_run/ (see
section 12). Additionally computes a per-timestep flood_timeseries.csv
(flooded_area_km2, flood_volume_m3) via compute_flood_timeseries_stats() —
a SEPARATE, independent computation from compute_flood_progression()'s
animation output (see "MESH-NATIVE ANIMATION" below for why they diverged)
— flood_volume_m3's max over the CSV is the physically correct peak total
flood volume (volume actually present in the domain at one real instant),
which is NOT the same as summing each pixel's own zsmax envelope maximum
(different pixels can peak at different times).

sfincs_subgrid.nc naming: identical default filename for both grid types.

QUADTREE GRID SUPPORT (rules 13, 14, 15, 16):
Enabled by default. Toggle: sfincs.grid.quadtree.enabled. VALIDATION:
quadtree.enabled requires subgrid.enabled = true (checked in 00_common.smk)
— subgrid is the only place a regular reference raster (dep_subgrid.tif)
exists for quadtree runs, and every postprocessing step that turns ugrid
data into a raster depends on it.

Refinement zones (workflow/src/quadtree_refinement.py,
build_refinement_polygons()):
  - River zone: river_network reaches buffered by width × river_buffer_factor
    (metric CRS via pick_utm_crs).
  - Coastal zone: land_polygons.gpkg coastline boundary buffered by
    coastal_buffer_m.
  - No overlay/priority logic needed — hydromt_sfincs's quadtree_grid
    refines each polygon's footprint independently to its own
    refinement_level, so a cell covered by both zones ends up at
    whichever level is higher regardless of GeoDataFrame row order.
  - Diagnostic plot: plot_refinement_zones() in src/plots.py →
    figs/01b_refinement_zones.png.

Any new diagnostic that reprojects a quadtree run's fine-resolution raster
output to WGS84 should use plots.py's _downsample_wgs_dataarray() helper —
see section 10. For a diagnostic driven by mesh-native data instead (e.g.
zs/zb, which stay mesh-native for animation — see "MESH-NATIVE ANIMATION"
in section 14), prefer plots.py's _mesh_overlay_setup() + da.ugrid.plot()
pattern (used by animate_flood_progression) over rasterizing first — it
plots actual mesh cell polygons via a PolyCollection, memory-scaled to cell
count rather than a dense raster.

_downsample_wgs_dataarray() — mirrors the file's numpy-array downsampling
convention (_PLOT_MAX_PX = 2,000,000 px) but for xarray DataArrays
reprojected to WGS84. Reprojecting a fine-resolution SFINCS run raster (e.g.
a quadtree run's subgrid, down to ~1.5 m pixels) to WGS84 for a whole delta
domain can produce tens of thousands of pixels per side; handing that
straight to imshow/FuncAnimation can exhaust memory (matplotlib allocates an
RGBA buffer sized to the full array). Wired into plot_max_inundation_map(),
plot_inundation_check(), and animate_flood_progression()'s regular-grid path
(its quadtree path is mesh-native — see "MESH-NATIVE ANIMATION" below — so
it never reprojects/rasterizes at all). Any new function that reprojects a
SFINCS output raster/DataArray to WGS84 for plotting should call this helper
right after the `.rio.reproject("EPSG:4326")` call.

IMPORT VERIFICATION NOTE: `ast.parse` and `snakemake -n` only catch syntax
errors and DAG-shape problems — neither catches a broken `from src.X import
Y` statement, since that is syntactically valid and only fails at actual
script-execution time (ModuleNotFoundError / AttributeError). After renaming
or deleting a src/ module, or moving a function between modules, verify with
an actual import-resolution check: parse every workflow/scripts +
workflow/src + tests file's `from src.X import Y` statements, import each
module X (with workflow/ on sys.path), and confirm Y is a real attribute. A
plain grep for the old module/function name is necessary but not sufficient
when the rename touches many call sites — it's easy to miss one.

  These are ALWAYS false positives — BASINS is defined in 00_common.smk via
  Snakemake include:, which the Python linter doesn't understand. The
  "snakemake is not defined" warning in every workflow/scripts/*.py file is
  the same kind of false positive — the `snakemake` object is injected by
  the Snakemake `script:` directive at runtime, not imported.
  GeoSeries.x / .y attribute errors in IDE are also false positives (pyright
  doesn't know the GeoSeries type).

RASTER ALIGNMENT:
  calculate_default_transform adds padding → extent > wgs84_bounds → visual
  misalignment. Fix: pass dst_bounds to _to_wgs84_grid() to use from_bounds.

MDT EXTRAPOLATION:
  MDT is NaN over land. Must call rasterio.fill.fillnodata() on the raw WGS84
  clip BEFORE resampling to UTM grid (applies to GEBCO's MDT correction in
  rule 05a; the DEM itself is never MDT-corrected, only geoid-offset
  corrected). Load with 5° margin (mdt_load_margin_deg) to ensure sufficient
  offshore reference pixels.

QUADTREE POSTPROCESSING MEMORY (fixed, three rounds -- basin 4267691, run_spinup):
  get_bed_level()'s subgrid reference raster (used to rasterize a quadtree
  mesh field onto a regular grid) can be large enough on its own, for a big
  domain, to blow past available memory -- not because the domain is
  unusually finely refined everywhere, but because the OLD mosaic logic
  applied the FINEST quadtree level's native pixel size across the WHOLE
  domain extent, even where only a small fraction of cells are actually
  refined that finely. Three rounds, each one layer deeper than the last:
  Round 1: compute_max_inundation -> hydromt_sfincs.utils.downscale_floodmap
    -> zsmax.ugrid.rasterize_like(dep) tried to allocate a (25180, 37500)
    float64 array (7+ GiB) and raised numpy ArrayMemoryError.
    Fix: src.postprocessing._coarsen_for_memory() (renamed from
    _coarsen_for_timeseries, which used to run only for animation time
    series under the wrong assumption a single frame always fits) now also
    runs inside compute_max_inundation with n_frames=1 whenever zsmax is a
    UgridDataArray (quadtree) -- coarsens the reference raster BEFORE it's
    used as the rasterize_like target, whenever the native resolution
    exceeds max_bytes (default 500 MB).
  Round 2: _coarsen_for_memory()'s OWN da_ref.coarsen(...).mean() call then
    failed too: "Unable to allocate 900. MiB for an array with shape (8393,
    3, 12500, 3) and data type bool". Cause: da_ref arrives already fully
    materialized (eager numpy-backed -- get_bed_level's rioxarray/
    data_catalog reads are not chunked); an un-chunked .coarsen().mean()
    with the default skipna=True needs to build nanmean's internal isnan
    mask (and other reduction bookkeeping) at a size comparable to da_ref
    itself, ON TOP of da_ref already resident -- enough on its own to
    exceed available memory even at "only" 900 MiB, since so little
    headroom remained after the ~7.5 GiB source array. Fix: da_ref.chunk({
    "y": 2000, "x": 2000}) before .coarsen(...).mean(), then .compute() only
    the final (already-small, coarsened) result.
  Round 3: even BEFORE either of the above ran, get_bed_level() ->
    _mosaic_quadtree_dep_levels() itself failed while BUILDING the mosaic:
    mosaic.where(~mosaic.isnull(), da.rio.reproject_match(mosaic)) tried to
    allocate a (25180, 37500) float32 array (3.5 GiB). Cause: the old
    implementation opened each dep_subgrid_lev*.tif at NATIVE resolution
    (rioxarray.open_rasterio without chunks reads eagerly) and reprojected
    every coarser level onto the FINEST level's native full-domain grid
    before combining -- so the huge array existed before either Round 1 or
    Round 2's fixes ever got a chance to run. Fix (the actual root-cause
    fix; Rounds 1-2 remain as defense-in-depth but are now mostly no-ops for
    the single-frame case): rewrote _mosaic_quadtree_dep_levels() to read
    each level directly at a shared, memory-bounded target resolution via
    plain rasterio (`ds.read(out_shape=..., resampling=Resampling.average)`
    -- a decimated read that never materializes the native-resolution array,
    unlike rioxarray's default eager open), computed from the same
    max_bytes-driven factor formula as _coarsen_for_memory. Levels are still
    combined finest-first (np.where(isnan(mosaic), coarser, mosaic)), so
    refined-zone detail still wins where present -- just built at a bounded
    resolution from the start instead of coarsening an already-huge array
    after the fact.
  If a similar ArrayMemoryError ever recurs in a NEW postprocessing/plotting
  function that touches get_bed_level()'s reference raster or the quadtree
  per-level dep files directly, check whether it's reading/building at
  NATIVE resolution somewhere upstream of any coarsening (Round 3's mistake)
  before assuming the existing _coarsen_for_memory() safety net will catch
  it -- that net only helps once the array is already in memory.

MESH-NATIVE ANIMATION (flood progression):
  hydromt_sfincs itself never rasterizes a quadtree mesh for PLOTTING —
  SfincsModel.plot_basemap() and xugrid's own da.ugrid.plot() render mesh
  data directly as a matplotlib PolyCollection built from the mesh's own
  cell polygons (memory scales with cell COUNT, not a rasterized pixel
  grid). The only rasterize_like() call anywhere in hydromt_sfincs is
  downscale_floodmap's genuine subgrid-resolution downscaling (a real
  analysis product, not a plotting convenience) — exactly the operation the
  quadtree-memory fixes above had to make memory-safe. Our own animate_
  flood_progression used to rasterize EVERY quadtree field before plotting,
  even though animations don't need subgrid-level pixel detail at all —
  purely a plotting convenience, unlike compute_max_inundation's actual
  downscaled analysis output.
  compute_flood_progression() (src/postprocessing.py) now returns h = clip
  (zs - zb, 0) mesh-native (xu.UgridDataArray, CRS restored via
  _ensure_ugrid_crs/mod.crs if missing) for a quadtree run, WITHOUT
  rasterizing and WITHOUT land/water masking (the land polygon overlay
  already shows the water boundary visually; sampling land-use per mesh
  cell centroid would be extra complexity for a purely cosmetic effect).
  plots.animate_flood_progression detects xu.UgridDataArray and renders via
  _mesh_overlay_setup() (reprojects land/river/domain overlays onto the
  MESH's own native CRS, e.g. UTM — not WGS84 like every raster-based plot
  in this module, since xugrid has no simple .rio.reproject() equivalent for
  a mesh's topology) + da.isel(time=0).ugrid.plot() building a PolyCollection
  once, updated per frame via collection.set_array(values.ravel()) — never
  rasterizes. Regular-grid runs are completely unaffected (same imshow path
  as before).
  VELOCITY ANIMATION WAS REMOVED ENTIRELY (out of scope for this project) —
  plots.animate_velocity, postprocessing.compute_velocity_timeseries,
  sfincs.sanity_checks.velocity_animation.enabled, the animation_velocity
  rule outputs, and storevel wiring in rules 13/14 are all gone. storevel is
  now hardcoded to 0 in both rule 13's and rule 14's sfincs.inp — nothing
  ever needs instantaneous u/v output anymore. Before removal, a candidate
  mesh-native rendering for velocity was investigated and rejected: SFINCS
  stores u/v on a SEPARATE staggered "interface/flux point" topology
  (SFINCS's own regular-grid output reader splits face vars like zs/zb from
  edge vars like u/v/qx/qy; the quadtree kernel's internal "uv point"
  bookkeeping is architecturally distinct from and not equal in count to the
  face/z-point set) — so u/v could not have been safely assumed to carry a
  usable Ugrid2d FACE mesh for direct mesh-native plotting the way zs/zb
  reliably do. Worth knowing if velocity output is ever needed again.
  Area/volume STATISTICS (flood_timeseries.csv, the sanity-check inundation
  ratio) deliberately did NOT move to mesh-native math, despite this being
  the obvious-looking next step: compute_max_inundation's da_hmax is already
  a UNIFORM-resolution raster (downscale_floodmap's output grid), so pixel-
  counting has no variable-cell-size bias to fix, and switching to summing
  raw per-cell mesh areas would mean losing the subgrid within-cell bed
  detail SFINCS subgrid modeling exists to capture (each quadtree cell would
  fall back to its own single zb value) -- a real accuracy downgrade for
  cells with significant internal bed variation (riverbanks, levees), not a
  neutral reformulation. Instead: compute_flood_timeseries_stats() (NEW
  function, src/postprocessing.py) decouples the CSV's per-timestep area/
  volume from compute_flood_progression()'s now-mesh-native animation output
  entirely -- processes ONE timestep at a time via hydromt_sfincs.utils.
  downscale_floodmap (same function/resolution compute_max_inundation uses)
  onto the SAME (already memory-bounded) reference grid, computes that
  frame's stats, discards it, moves on. This decouples memory cost from
  frame count entirely (previously, the whole multi-frame series had to be
  rasterized onto the reference grid at once, forcing a coarser max_bytes
  budget to fit ALL frames simultaneously) — so stats now use
  STATS_MAX_BYTES (1.5 GB, single-frame) instead of animation's shared
  500 MB default, meaning genuinely MORE accurate stats than before, not
  just a memory fix. downscale_floodmap is called with hmin=0.0 (not
  threshold_m) so flood_volume_m3 sums ALL positive depth present in the
  domain, matching a literal "total flood volume" -- threshold_m is applied
  separately, only to decide which pixels count towards flooded_area_km2.
  Round 4 (basin 4267691, run_event): compute_flood_timeseries_stats's own
  get_bed_level -> _mosaic_quadtree_dep_levels crashed AGAIN, now inside
  mosaic.rio.write_crs(...).rio.write_transform(...) — "Unable to allocate
  1.76 GiB" for an array already sized right at STATS_MAX_BYTES. Cause:
  rioxarray's write_crs()/write_transform() deep-copy the WHOLE array by
  default (inplace=False) even though they only touch metadata — chaining
  two out-of-place calls on an array already at the memory budget needs
  multiple simultaneous full-size copies. Fix: pass inplace=True to both
  (in _mosaic_quadtree_dep_levels AND _coarsen_for_memory's own write_crs
  call) — sets metadata directly, no copy. If a similar crash ever recurs
  right at a `.rio.write_*()` call specifically (not the surrounding
  compute), check for a missing inplace=True first.
  _rasterize_like() and _n_frames() (formerly used by compute_velocity_
  timeseries and the old rasterizing compute_flood_progression) were
  removed as dead code once nothing called them anymore — the ONLY
  remaining raster-projection path in postprocessing.py is
  hydromt_sfincs.utils.downscale_floodmap (compute_max_inundation,
  compute_flood_timeseries_stats), which correctly downscales BOTH grid
  types (unlike _rasterize_like, which was a no-op for a regular grid and
  would have silently skipped the coarse-cell-to-fine-subgrid-pixel
  refinement a regular grid needs too, not just quadtree).
  _coarsen_for_memory()'s n_frames parameter was also dropped (simplified
  to always assume a single frame) once every remaining caller passed 1 —
  velocity's multi-frame usage was the only caller that ever passed
  anything else.

RUNNING A SINGLE BASIN:
  BASINS (00_common.smk) is auto-discovered from the delta_polygons dataset
  and not directly configurable in config.yml, but can be restricted per-run
  via an optional CLI-only --config override, target_basins (a list of
  basin_id ints). If unset, BASINS is unchanged (all discovered basins) —
  fully backward compatible. Works with the existing named targets
  (preprocess/build/all) directly, no per-file target list needed:
    snakemake build --cores N --config target_basins="[2433835]" --forceall
  --forceall forces every job for that basin to re-execute even if its
  outputs look up to date; drop it to only rerun what's missing/stale.
  Unknown basin_ids raise a ValueError at config-parse time (before the DAG
  is built) rather than silently producing an empty/wrong DAG.

SNAKEMAKE OUTPUT SYNC:
  Always keep _PREPROCESS_OUTPUTS / _BUILD_OUTPUTS in Snakefile in sync with
  rule outputs, including the config-conditional blocks (river conditioning,
  subgrid, quadtree, spinup). MissingInputException for tracked outputs
  almost always means one of these lists is stale. Validate with
  `snakemake -n` after any rule/output change.

SNAKEMAKE FAILURE CLEANUP:
  Snakemake's failure-cleanup deletes ALL declared outputs of a failed job,
  including any raw simulation output already written before the failure
  (e.g. sfincs_map.nc, the .rst restart file from rule 14) even if the
  underlying SFINCS run itself completed successfully. There is no way to
  recover a crashed job's outputs without re-running the failed rule; only
  upstream rules whose own outputs are already on disk and untouched can be
  skipped.

  tools/sync_environment.py + the local "sync-environment-yml" pre-commit
  hook (added to .pre-commit-config.yaml) keep environment.yml in sync with
  the live hmt_sfincs_dev conda env automatically: only fires when
  environment.yml is already part of a commit (files: ^environment\.yml$),
  regenerates via `conda env export -n hmt_sfincs_dev`, strips the trailing
  prefix line, and blocks+rewrites the commit if stale (same pattern as
  ruff --fix). Verified against the real environment: it correctly caught
  drift (several packages installed since the last static export —

  restored from a deliberately-corrupted file.

  hooks outside an activated conda shell (e.g. from an IDE's git
